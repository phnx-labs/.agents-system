Self-healing bookkeeping for
remote --host runs

A remote agent survives a dropped connection — but today the local record doesn't track that. Three status bugs leave tasks frozen or falsely failed. This plan makes the local view reconcile itself against remote ground truth, with no daemon and no new connection to hold open.

01The three bookkeeping bugs

Remote runs launch detached on the host (nohup … &) and write their output to a log plus an exit code to <id>.exit. The agent therefore outlives any dropped connection. The local task record is where reality slips.

SymptomRoot cause (file:line)
BUG 1 Kill the local run (sleep, Ctrl-C) → agents hosts ps shows running forever. Terminal status is written only after followHostTask returns — dispatch.ts:85-96. Killed follower ⇒ never runs. listTasks() reads verbatim — tasks.ts:67-80.
BUG 2 Reattach with logs -f, watch it finish → still shows running. doLogs gets the exit code back but never calls updateTask — hosts.ts:196-197.
BUG 3 A run longer than the 1 h follow window exits 1 locally — a healthy job looks failed. followHostTask returns -1 on deadline (progress.ts:33,56); exec.ts:419 maps -1 → exit 1; dispatch.ts:92 freezes it at unknown.

02How a remote run actually works

The offload is deliberately connectionless. SSH is used only to kick off the run and to poll a file — never to hold a live pipe to the agent.

03The design choice that matters ★

You asked the sharp question: why not hold an open channel and let everyone watch it, instead of writing to a file and polling? Here are the two models side by side. The short answer: the file model is the "everyone watching" model — done so it survives the watchers leaving.

PropertyA · held channel + hubB · file + reconcilewhy it matters

the takeaway "Hold a channel so everyone can watch" sounds like it adds observers. It actually centralizes them behind a crashable hub and ties live state to a connection — the one thing that must not die. Model B is the decentralized version of the same idea: every eye is independent, and the truth outlives all of them. The bookkeeping fix is just teaching the local record to re-read that truth when someone looks.

04The fix — reconcile-on-read

Events are rare (a run finishes once) and watchers are intermittent, so we don't push over a held channel — we cheaply re-read .exit whenever someone runs ps or logs. One pure mapping + one small network module, wired into the paths that already exist.

The two new pieces

// tasks.ts — pure, no SSH; a real remote exit code is never -1
export function terminalPatch(code) {
  return { status: code === 0 ? 'completed' : 'failed', exitCode: code, finishedAt: iso() }
}

// reconcile.ts (new) — the one place ps + logs both route through
export function readRemoteExit(target, remoteExit) {
  const r = sshExec(target, `cat ${remoteExit} 2>/dev/null`, { timeoutMs: 6000, multiplex: true })
  if (r.timedOut || r.code === null || r.code === 255) return { state: 'unreachable' }  // host down
  const out = r.stdout.trim()
  if (out === '') return { state: 'running' }        // .exit absent or mid-write
  return { state: 'done', code: parseInt(out, 10) || 0 }
}
the semantic fix (bug 3) -1 from followHostTask means "local follow window closed; the remote run continues" — never a failure and never terminal. Everywhere it's consumed we now treat it as still running, resolve later: dispatch leaves the record running; exec.ts:419 and doLogs exit 0 (the reattach hint is already printed at progress.ts:57).

06Edge cases, guardrails & delta spec

Edge cases

  • Host offline → unreachable, left running. Never false-failed.
  • .exit absent vs empty → both "still running"; only an integer transitions.
  • Concurrent reconciles → idempotent (same .exit ⇒ same patch); no locking.
  • Timeout on reattach → no-op, stays running, a later ps finishes it.
  • .exit lost after finish → stays running; strictly better than a false fail.

Deliberately NOT doing scope

  • No daemon / background watcher.
  • No async rewrite of ssh-exec.ts for parallel reconcile.
  • No file locks (idempotence makes them moot).
  • No PID-liveness heuristics for lost .exit.
  • Don't delete the now-vestigial unknown status.

Delta spec — the contract after this change

For any HostTask, status:'running' means "not yet confirmed finished." It is resolved to completed/failed by any read (ps, logs) that can reach the host and find a populated .exit. An unreachable host or absent/empty .exit never changes status. followHostTask returning -1 is "detached, still running" (shell exit 0), never a failure. terminalPatch is the sole exit→status authority.

verification Two independent planners (read-only, blind to this draft) reproduced all three bugs at the same file:line and independently arrived at this architecture. The one divergence — sequential vs parallel reconcile — resolved to host-deduped sequential: one sshReachable probe per host keeps ps responsive without an async rewrite.
